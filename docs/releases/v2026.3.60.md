# v2026.3.60

**Release date:** 2026-03-06

## Telemetry polling and optional get_controller

### Added

- **Telemetry polling (get_device_msg keepalive)** (#79)
  - `start_polling(interval_seconds=10, interval_when_active_seconds=1, acquire_controller=False)` — periodic get_device_msg; 1s when robot active
  - `stop_polling()` / `is_polling` — lifecycle
  - `get_status(acquire_controller=False)` — single snapshot via get_device_msg (no get_controller by default)
  - Auto-start polling after ~5s with no telemetry when using `watch_telemetry()`
  - Polling interval 1–3600s; default 10s idle, 1s when active

- **Optional get_controller for telemetry**
  - Telemetry (get_status, polling) does not call get_controller by default — mobile app can stay in control
  - Use `acquire_controller=True` or `get_controller()` only when sending commands (lights, buzzer, plans, etc.)
  - Robot responds to get_device_msg without controller; tested with app in control

- **Docs and scripts**
  - `docs/telemetry_mqtt_and_brokers.md` — get_device_msg response, MQTT semantics, two-broker test, coexistence with app, “Why didn’t HASS get updates?”
  - `docs/telemetry_polling_measurement.md` — measured behaviour (no DeviceMSG stream after one get_device_msg)
  - `scripts/monitor_telemetry_after_get_device_msg.py` — measure traffic after one get_device_msg
  - `scripts/test_two_brokers_telemetry.py` — send on one broker, subscribe on both
  - `scripts/test_polling_with_app_in_control.py` — test polling with/without get_controller; response counts
  - `scripts/check_telemetry_connection.py` — quick connectivity check

### Changed

- `get_status()` — no longer calls get_controller by default; use `acquire_controller=True` to restore previous behaviour
- `start_polling()` — no longer calls get_controller by default; use `acquire_controller=True` if needed
- POLLING_INTERVAL_MIN = 1 (was 5)

### Fixed

- HASS coexistence: telemetry-only use no longer takes control from the mobile app
